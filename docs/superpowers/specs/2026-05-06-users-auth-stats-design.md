# Users, Auth, and Stats — Design

**Date:** 2026-05-06
**Status:** Approved (pending spec review)
**Phase:** A (login + per-race stats; no ELO, no multiplayer)

---

## 1. Goal

Add user accounts and per-race stats tracking to Arithmetic Racer, while keeping anonymous play frictionless. ELO and multiplayer are explicitly out of scope and will be designed separately in Phase B.

**In scope:**
- Sign up / sign in (Google OAuth + email/password)
- Password reset via email (no email verification gate)
- Per-race stats persistence
- Personal profile screen with aggregates and recent races
- Anonymous → registered race-history claim on signup
- Username-pick flow with offensive-name screening

**Explicitly out of scope:**
- ELO / ratings of any kind
- Multiplayer (Phase B)
- Public profiles, leaderboards, user search
- Avatars, badges, change-email, account deletion (roadmap, not v1)
- Email verification (only a non-blocking welcome email)
- Magic-link login

---

## 2. Stack decision

**All-Cloudflare.**

| Layer | Tech |
|-------|------|
| Static frontend | Workers static assets (already configured in `wrangler.jsonc`) |
| API | Cloudflare Worker (same Worker, mounted alongside static assets) |
| Auth library | `better-auth` with D1 adapter |
| Identity DB | D1 (SQLite) |
| Sessions | better-auth, HTTP-only cookies, D1-backed |
| Email (password reset + welcome) | Resend |
| Username profanity check | `obscenity` npm package + static deny list |

**Why this over Supabase + CF (the close runner-up):**
- One vendor, one dashboard, one deploy command.
- Sets up Phase B (multiplayer) cleanly: Durable Objects are purpose-built for real-time race rooms, and DO will share auth context with the same Worker.
- The Supabase advantage was easier auth (~1-2 hr setup delta) — meaningful but not stack-deciding.

**Why this over Render + Postgres + Express (the original plan):**
- Render free tier sleeps; cold starts hurt UX.
- Workers run at the edge with no cold start.
- Original plan predates the multiplayer architecture; DO is a strictly better primitive for race rooms than Socket.io + Redis.

## 3. Architecture

```
[ Browser ]
    |
    | HTTPS (same origin — no CORS)
    v
[ Cloudflare Worker ]
    |---- static assets (public/)
    |---- /api/auth/*       --> better-auth --> D1 (user, account, session, verification)
    |---- /api/race-result  --> Worker code  --> D1 (race_results)
    |---- /api/me*          --> Worker code  --> D1
    |---- /api/stats/by-device/:id  --> D1 (anon stats for header pills)
                                  |
                                  +--> Resend (welcome + password reset)
```

**Routes:**
- `/api/auth/*` — owned by `better-auth`: sign-up, sign-in, OAuth callback, password reset, sign-out, session check.
- `POST /api/race-result` — persist a race result. Body: `{device_id, difficulty, finished, finish_time_ms, problems_correct, problems_attempted, longest_streak, ...}`. If session cookie present, sets `user_id`. Else `user_id` is NULL.
- `GET /api/me` — own profile (aggregates + last 10 races). Requires session.
- `POST /api/me/username` — rename. Validates uniqueness + offensive content. Requires session.
- `GET /api/stats/by-device/:device_id` — anon header stats (best time, total races) for a localStorage device.
- Anything else → static asset handler.

**Sessions:** HTTP-only, Secure, SameSite=Lax cookie. No tokens in localStorage (XSS-safer). Browser sends cookie automatically with `fetch` calls to `/api/*`.

**Anonymous device ID:** UUID v4 generated lazily on first race start, stored in `localStorage.deviceId`. Sent in race-result POSTs for both anon and logged-in users (logged-in tracking is for analytics and the claim flow).

## 4. Database schema

Migrations live in `migrations/` (the directory already exists, currently empty).

### 4.1 `better-auth`-managed tables

Generated by running `npx @better-auth/cli generate` against the project's better-auth config. Output is committed as a numbered migration file.

- `user` — id, email, email_verified, name, image, created_at, updated_at
- `account` — links a user to credentials (Google account or password hash)
- `session` — active sessions, referenced by HTTP-only cookie
- `verification` — short-lived tokens for password reset and welcome confirmation

### 4.2 Custom field on `user`

Added in a follow-up migration:

- `username` (text, UNIQUE, NOT NULL) — public display name. Validated at signup with the static deny list + `obscenity`.

### 4.3 `race_results` table (custom)

| Column | Type | Notes |
|--------|------|-------|
| `id` | text | UUID, PK |
| `user_id` | text | FK→user.id, NULL for anonymous races |
| `device_id` | text NOT NULL | localStorage UUID, set for both anon and logged-in |
| `difficulty` | text NOT NULL | CHECK(`easy` \| `medium` \| `hard`) |
| `finished` | boolean NOT NULL | false if user quit |
| `finish_time_ms` | integer | NULL if didn't finish |
| `problems_total` | integer NOT NULL DEFAULT 20 | |
| `problems_correct` | integer NOT NULL | |
| `problems_attempted` | integer NOT NULL | |
| `avg_time_per_problem_ms` | integer NOT NULL | denormalized for read performance |
| `accuracy_pct` | real NOT NULL | denormalized |
| `longest_streak` | integer NOT NULL | |
| `played_at` | timestamp NOT NULL | indexed |

**Indexes:**
- `(user_id, played_at DESC)` — profile aggregates and recent-races list
- Partial index on `(device_id) WHERE user_id IS NULL` — for the claim-on-signup query

### 4.4 Key queries

**Claim on signup:**
```sql
UPDATE race_results
SET user_id = ?
WHERE user_id IS NULL AND device_id = ?;
```

**Profile aggregates (per difficulty):**
```sql
SELECT difficulty,
       COUNT(*) AS races_played,
       COUNT(CASE WHEN finished THEN 1 END) AS races_finished,
       MIN(CASE WHEN finished THEN finish_time_ms END) AS best_time_ms,
       AVG(accuracy_pct) AS avg_accuracy,
       AVG(avg_time_per_problem_ms) AS avg_problem_time_ms
FROM race_results
WHERE user_id = ?
GROUP BY difficulty;
```

**Recent races (with computed race number):**
```sql
SELECT *,
       ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY played_at) AS race_seq
FROM race_results
WHERE user_id = ?
ORDER BY played_at DESC
LIMIT 10;
```

**Schema-evolution guidance:** keep `race_results` columns narrow and additive. To compute new stats later, add columns; don't repurpose existing ones. If a definition changes (e.g., accuracy formula), add `accuracy_pct_v2` rather than mutating in place.

## 5. Auth flows

### 5.1 Anonymous play

1. On first race, generate `device_id` UUID → store in `localStorage.deviceId`.
2. Race finishes → POST `/api/race-result` with `{device_id, ...stats}`. `user_id` NULL.
3. No login required to play.

### 5.2 Sign up — email/password

1. User clicks `CREATE ACCOUNT` in header → modal opens.
2. Form: email, **confirm email** (typo guard), password (min 8 chars), username.
3. Client-side: inline username preview (`✓ Available` / `✗ Taken` / `✗ Not allowed`).
4. Submit → better-auth's email/password sign-up endpoint (exact path determined by library config, mounted under `/api/auth/`).
5. Server validates: `obscenity` + deny-list on username; uniqueness on email and username.
6. On success: better-auth creates `user`+`account`+`session`, sets HTTP-only cookie.
7. **Claim step:** Worker runs the claim query with the new `user_id` and the `device_id` from the request body.
8. Worker fires welcome email via Resend (passive verification — surfaces typos).
9. Modal closes; header re-renders into logged-in state.

### 5.3 Sign up — Google OAuth

1. Click `Continue with Google` → redirect to Google → returns to `/api/auth/callback/google`.
2. better-auth creates user (email from Google profile, no password). Username is empty.
3. **Username gate:** if user has no username, show one-field "Pick your username" modal. Cannot dismiss until set. Same validation as email/password signup.
4. Claim step runs after username is set.
5. No welcome email (Google already verified the address).

### 5.4 Login — email/password

Standard flow, owned by better-auth. Session cookie set on success. Claim step does **not** run on login (only on signup) — assumes any earlier claims happened at signup time.

### 5.5 Login — Google

Same as Google signup; better-auth dedupes by email if the user already exists.

### 5.6 Forgot password

1. "Forgot password?" link in login modal → form: email.
2. Submit → better-auth issues `verification` token, fires email via Resend with link to `/reset-password.html?token=…`.
3. Reset page → new password form → POST to better-auth → token verified → password updated → session created → redirect home.

### 5.7 Logout

`POST /api/auth/sign-out` → cookie cleared. `localStorage.deviceId` is **not** cleared — user stays linked to that device for any anonymous races they play before logging back in.

### 5.8 Username rename

1. Profile screen → "Change username" → modal input.
2. Same validator as signup.
3. POST `/api/me/username` → uniqueness check → update.
4. No history rewriting; `race_results` does not store username.

### 5.9 Anonymous → registered claim semantics

- Claim runs only at signup. After claim, that `device_id`'s prior anon races have `user_id` set.
- If user logs out and plays anonymously on the same device, those new anon races again have `user_id` NULL until a future signup claims them.
- If a different person signs up on the same device (shared family laptop), they claim the new anon races. Already-claimed races are not touched (the WHERE clause filters by `user_id IS NULL`). Acceptable edge cases for v1.

## 6. UI changes

### 6.1 Header (always visible)

**Logged out:**
```
[avatar] {generated_handle}  [CREATE ACCOUNT]  [SIGN IN]  [Best 0:48.1M]  [12 Races]
```
- Display name is the generated handle (`BraveOtter`), not "Guest" — anon stats *are* tracked by device, so the handle has continuity.
- `CREATE ACCOUNT` is the primary CTA (bright color, prominent).
- `SIGN IN` is secondary.
- Stat pills populated from `GET /api/stats/by-device/:id`.
- `Best` pill shows time + difficulty letter (e.g., `0:48.1M`).

**Logged in:**
```
[avatar] {username} ▾  [Best 0:48.1M]  [12 Races]  [⚙]  [↪logout]
```
- Username dropdown → Profile, Log out.
- Stat pills from `GET /api/me`.

### 6.2 Auth modal

Single component, two tabs (Log in / Sign up). Top of modal: prominent `Continue with Google`.

- Sign up tab: email, confirm email, password, username, with inline validation.
- Log in tab: email, password, "Forgot password?" link.
- Errors shown inline, not as alerts.

### 6.3 Pick-username modal (Google OAuth only)

Single field, blocks the lobby until set. Same validator as signup form.

### 6.4 Profile screen (`#profile` section)

**Header card:**
- Avatar placeholder + username + small edit pencil → opens rename modal.
- Right-aligned headline: `2.4s avg` (avg time per problem across all races).

**Stat tiles row (6 tiles):**
1. Best Easy
2. Best Medium
3. Best Hard
4. Total Races
5. Overall Accuracy
6. Finish Rate (% completed without quitting)

**Account info box (left) and avatar slot (right):**
- Left: Racing Since (signup date), Email (plain display in v1 — verification indicator is roadmap).
- Right: "Change Avatar" button — disabled v1 placeholder.

**Latest Race Results table:**

| Race # | Difficulty | Time | Accuracy | Avg/problem | Date |
|--------|-----------|------|----------|-------------|------|
| 12 | Hard | 1:12.0 | 84% | 3.6s | 2h ago |

Race # is computed via `ROW_NUMBER()` at read time — no per-user sequence column.

### 6.5 Reset password page (`/reset-password.html`)

Single form: new password + confirm. POSTs to better-auth's reset endpoint with token from URL.

### 6.6 Existing screens

- `#lobby` — unchanged.
- `#race` — lane label uses logged-in username if available, else generated handle.
- `#results` — when logged in: "Saved to your profile ✓". Otherwise: "Sign up to track your stats" CTA opens auth modal.

### 6.7 File layout

**Frontend additions:**
```
public/
├── index.html              (+ header, + modal shells, + #profile section)
├── style-a.css             (+ header, modals, profile, pills)
├── reset-password.html     (new)
└── src/
    ├── auth.js             (new — wraps better-auth client SDK)
    ├── stats-api.js        (new — POST race-result, GET /api/me, GET /api/stats/by-device/:id)
    ├── profile.js          (new — renders profile screen)
    ├── username-validator-client.js  (new — same validator runs client-side for inline preview)
    ├── ui.js               (existing — wire header + modal triggers)
    └── runner.js           (existing — POST race-result on race finish)
```

**Backend (new):**
```
worker/
├── index.js                (Worker entry; routes /api/* + falls through to assets)
├── auth.js                 (better-auth config: providers, D1 adapter, Resend hooks)
├── routes/
│   ├── race-result.js      (POST handler + claim-on-signup helper)
│   └── me.js               (GET /api/me, POST /api/me/username, GET /api/stats/by-device/:id)
├── username-validator.js   (deny list + obscenity wrapper — single source of truth, imported from client mirror)
└── email.js                (Resend wrapper for welcome + reset templates)
```

**`wrangler.jsonc`:** add `main: "worker/index.js"`, D1 binding, env vars (Google OAuth client ID/secret, Resend API key, BETTER_AUTH_SECRET) via `wrangler secret`, not committed.

## 7. Testing & deployment

### 7.1 Unit tests

- `username-validator.test.js` — banned terms, leetspeak bypasses, reserved words (`admin`, `moderator`, `system`), valid names, length boundaries.
- Existing `game.test.js`, `bot.test.js`, `handles.test.js` — unchanged.

### 7.2 Worker route integration tests

Use `vitest-pool-workers` with Miniflare and an in-memory D1.

- Anon `POST /api/race-result` persists with `user_id` NULL.
- Logged-in `POST /api/race-result` persists with `user_id` set; claim step runs only on signup, not on each POST.
- Signup runs claim: prior anon races for that `device_id` get the new `user_id`.
- `GET /api/me` aggregates per difficulty, returns last 10 races.
- `POST /api/me/username` rejects banned, taken, and self-collision; accepts valid.
- `GET /api/stats/by-device/:id` returns anon-only stats (does not leak claimed races).

### 7.3 Manual end-to-end

Auth library code paths are well-tested by better-auth. We verify *config* manually on `wrangler dev`:
- Google OAuth callback URL works (local + prod).
- Welcome email and password-reset email arrive via Resend.
- Session cookie is set, browser sends it on `/api/*` calls.
- Reset link works, sets new password, signs user in.

### 7.4 Deployment

- D1 database created once: `wrangler d1 create arithmetic-racer`.
- Migrations applied via `wrangler d1 migrations apply`.
- Secrets set via `wrangler secret put`: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `RESEND_API_KEY`, `BETTER_AUTH_SECRET`.
- Single `wrangler deploy` for the whole app (frontend + Worker + bindings).

## 8. Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| Google OAuth setup friction (project, redirect URIs for local + prod) | Document exact steps in implementation plan; test on `wrangler dev` first. |
| Resend free tier limits (100/day in dev mode without verified domain) | Verify domain when traffic warrants. v1 should never approach the limit. |
| D1 5 GB free cap (race_results ~200 bytes/row → ~25M rows) | Non-issue for years. |
| `obscenity` false positives (Scunthorpe-style substring matches) | Use library's match-severity tuning; manual deny list curation; mailto appeals path post-launch. |
| Anon→registered claim race condition (signup completes mid-race) | Claim runs in same Worker invocation as session creation; frontend POSTs race result only after race ends; window is small. |
| `wrangler.jsonc` currently only configures static assets | Implementation plan includes the explicit `main` + `d1_databases` additions. |
| User typos email at signup, locked out later on password reset | Confirm-email field + welcome email surface typos before they matter. |

## 9. Future work (not v1, captured for roadmap)

- **Phase B — multiplayer + ELO:** real-time race rooms backed by Durable Objects; server-authoritative game state; ELO updates on finish. Designed separately.
- **Account management additions:** change avatar, change email (with re-verification), download race data (CSV/JSON export), delete account (target v1.1 for trust).
- **Public surfaces:** public profiles at `/u/<username>`, best-times leaderboards, user search.
- **Detailed per-problem logging:** problem text, user's answer, time-to-answer, correct/incorrect — enables "weakness analysis" and replays.
- **Email verification gate** (only if abuse pressure ever emerges).
- **Magic-link login** (deferred indefinitely per user preference).

## 10. Summary of decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| Hosting platform | All-Cloudflare | One vendor, edge-native, sets up Phase B (Durable Objects) cleanly |
| Identity DB | D1 (SQLite) | Single-platform, generous free tier, plays with existing `migrations/` |
| Auth library | better-auth | Closest Supabase-equivalent dev experience on Workers; D1 adapter |
| Login methods | Google OAuth + email/password | Magic link adds friction without enough payoff |
| Email provider | Resend | Free tier, 10-min CF integration, used only for welcome + reset |
| Email verification | Not gated; passive welcome email | No high-value abuse vector for a math game |
| Username strategy | Pick at signup, must be unique | User chose this over auto-generate |
| Username screening | Static deny list + `obscenity` | Catches obfuscation tricks; runs in-Worker; no external API |
| ELO timing | Deferred to Phase B | Needs separate design; doesn't make sense without PvP |
| Stats granularity | Standard (per-race aggregates, no per-problem log) | Profile gets everything it needs without volume |
| Anon → registered | Tracked by device_id, claimed on signup | User chose this for better UX |
| Public surfaces | None in Phase A | Defer until ELO ships |
